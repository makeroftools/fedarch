> ### CCC‑ID generation ONLY in CCC workspace — tools + ADMIN = NEVER

| Field | Value |
|-------|-------|
| **Rule ID** | R-194 |
| **Status** | 🔒 **IMMUTABLE** |

### Full Rule Text (per SharedKernel §Core Rules)

```
CCC‑ID generation ONLY in CCC workspace — tools + ADMIN = NEVER
```

### Plain Language Translation

| Role | Workspace | May Generate CCC‑IDs? |
|------|-----------|:---------------------:|
| You are a DEFAULT user (u‑<ccc>_user) | **CCC** | ✅ **YES** |
| You are a DEFAULT user (u‑<ccc>_user) | tools | ❌ NO |
| You are a DEFAULT user (u‑<ccc>_user) | ADMIN | ❌ NO |
| You are an ADMIN user (a‑<ccc>_dev) | ANY workspace | ❌ **NEVER** |
| You are a TOOL AGENT (t‑<tool>_tool) | ANY workspace | ❌ **NEVER** |

> **Bottom line:** Only DEFAULT users in the CCC workspace may generate new CCC‑IDs. Everyone else, everywhere else — reference only.

## 🎯 Purpose & Rationale

### Why This Rule Exists

| Reason | Detail |
|--------|--------|
| **Attribution Integrity** | CCC‑IDs MUST trace to a SINGLE contributor in a SINGLE workspace. Generation from non‑CCC workspaces breaks the audit trail. |
| **Sequence Consistency** | Only the CCC workspace maintains the sequential counter per contributor per week. Non‑CCC workspaces would create duplicate or out‑of‑sequence IDs. |
| **Separation of Concerns** | CCC workspace = production (user interaction, CCC‑ID generation). tools workspace = strategy, META, MAIT (no generation). ADMIN workspace = configuration (no generation). |
| **Immutable Governance** | R-194 is 🔒 IMMUTABLE — it cannot be overridden or bypassed. Any attempt to do so is an automatic #BadAgent. |

### Design Principles

| Principle | How R-194 Supports It |
|-----------|----------------------|
| **Traceability** | Every CCC‑ID is created in the workspace designed for attribution |
| **Consistency** | Single workspace = single sequence per contributor per week |
| **Security** | Non‑CCC workspaces CANNOT pollute the CCC‑ID sequence |
| **Simplicity** | One rule covers ALL non‑CCC workspaces and ALL non‑DEFAULT users |

## 📋 R-194 / R‑206: Complementary Axes

**Calhoun 🎖️ VSA Finding (verified via SK D‑050):** R-194 and R‑206 are **co‑equal IMMUTABLE complementary orthogonal axes** — NOT hierarchical.

| Rule | Axis | Scope |
|------|------|-------|
| **R-194** | **WORKSPACE** | CCC‑ID only in CCC workspace (tools/ADMIN/events/P.O.P./any = NEVER) |
| **R‑206** | **ROLE/USERNAME** | ADMIN accounts (a‑<ccc>_dev) NEVER generate; reserved for DEFAULT users (u‑<ccc>_user) ONLY |

> **Both MUST pass for CCC‑ID generation:** DEFAULT user (`u‑<ccc>_user`, per R‑206) **AND** CCC workspace (per R-194). Neither supersedes the other. This doc presents them as complementary, not hierarchical.

### Decision Tree (Updated with Complementary Axes)

         ├── YES → ✅ MAY generate CCC‑ID. (R-194 ✅ + R‑206 ✅)
         └── NO  → ❌ NEVER generate. (R-194 ❌)
